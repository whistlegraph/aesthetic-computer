Can This Contract Support Other Sigils?

Short answer: yes, contract-level. The current FA2 contract stores bytes fields and does not hard-code a $-only token namespace.

Current limitation is app-level: the keeps pipeline, validations, and index source are intentionally KidLisp-focused right now.

To support !tapes, *clocks, #paintings on this same contract:
  1. Normalize piece identity as sigil + id in mint payloads and content-hash keys.
  2. Generalize ownership checks to each source domain (KidLisp, tapes, clocks, paintings).
  3. Add per-sigil metadata templates and media generation paths.
  4. Expose mixed-sigil indexing/filtering in UI without losing per-handle ownership guarantees.

Keeps Process

  1. Pick: choose one of your pieces from Unkept.
  2. Prepare: validate ownership, bake media, bundle HTML, and pin to IPFS.
  3. Review: confirm fee + artifact links in the Keep modal.
  4. Sign: wallet signs and sends keep to the active contract.
  5. Confirm: chain confirms; index updates; marketplaces ingest token metadata.
Rekeep path: use Rekeep to regenerate media + metadata for an already-kept token.

Deeper Notes

A Keep is a KidLisp piece turned into a verifiable on-chain collectible. The mint points to permanent media on IPFS and records ownership in the active Tezos contract.

Three layers work together:
  1. Contract layer (Tezos): token id, owner, transfer history, and keep fee payment are finalized on-chain.
  2. Media layer (IPFS): HTML artifact, thumbnail, and metadata JSON are content-addressed.
  3. Index layer (keeps.kidlisp.com): human-readable search and status views for artists and collectors.
What this guarantees:
  • Unique mapping on this contract from KidLisp code to token id (via on-chain content hash registry).
  • Clear provenance: mint transaction, first minter, creator mapping, and token history are public.
  • Collector ownership follows FA2 ledger state and marketplace transfers.
What can still change:
  • Token metadata may be refreshed by authorized flows until metadata is explicitly locked for that token.
  • UI indexing and presentation can improve without changing token ownership or transaction history.

Intent as of March 9, 2026: v9 is the long-lived canonical KidLisp Keeps contract.

Longevity + Rollover Strategy

Target is no rollover. Like Bitcoin-era stability, the default plan is to keep one canonical production contract for as long as it is secure and functional.

Rollover is only considered if at least one trigger occurs:
  1. Critical security issue in production logic.
  2. Protocol/runtime incompatibility that breaks normal keep, transfer, or metadata operations.
  3. Irreversible bug causing unsafe ownership, payout, or token integrity behavior.
  4. Required capability cannot be added safely within existing contract constraints.
If rollover is ever needed, process is:
  1. Publish a public notice and rationale before deployment.
  2. Keep old contract live as historical record (readable, verifiable, and linked in About history).
  3. Deploy successor contract with explicit versioning and documented differences.
  4. Update keeps.kidlisp.com to clearly mark Active vs Legacy contracts.
  5. Never erase history: old tokens remain valid and traceable to their original contract.

Policy: no silent migrations, no hidden contract switches, and preserved provenance across generations.

\section{Introduction}

Creative coding platforms are typically evaluated at the system level: language design, editor affordances, community size~\citep{reas2007processing,mccarthy2015p5js,resnick2009scratch}. This paper takes a different approach. We study one piece---\texttt{notepat}---inside \ac{}, and ask how a single interactive program can reshape the platform that hosts it.

\texttt{notepat} is a melodic keyboard instrument. It is directly addressable as \texttt{aesthetic.computer/notepat} and through the branded domain \texttt{notepat.com}. As of March 2026, it is the most complex piece in the system, the boot piece on bare-metal hardware, and the primary subject of performance testing infrastructure. None of this was planned at the outset. The instrument grew, and the platform grew around it.

\section{Origins (June--July 2024)}

The first commit appears on June~27, 2024, under the name \texttt{notepad}---a typo that was later reinterpreted as a portmanteau of ``note'' and ``pad.'' The initial version was a minimal keyboard sampler: QWERTY keys mapped to chromatic notes, a single sine waveform, no UI beyond colored rectangles for active keys.

Within five days the piece had gained:
\begin{itemize}
  \item Synth fade-out (25ms linear decay to prevent clicks),
  \item Documentation references in the project index,
  \item The corrected name \texttt{notepat}.
\end{itemize}

At this stage the piece was roughly 200 lines. It used the standard \ac{} piece lifecycle (\texttt{boot}, \texttt{paint}, \texttt{act}) and the platform's built-in \texttt{sound.synth} API.

\section{Feature Growth}

\subsection{Audio engine (2024--2025)}

The synthesis surface expanded across several axes:

\begin{itemize}
  \item \textbf{Waveforms}: sine $\rightarrow$ triangle, sawtooth, square, noise (selectable via keyboard shortcut or parameter).
  \item \textbf{Room reverb}: Three-tap delay line (120\,ms intervals), 0.55 feedback coefficient, wet/dry mix controllable via trackpad X-axis slider.
  \item \textbf{Pitch shift}: Continuous $\pm$1 octave via trackpad Y-axis, with per-sample smoothing on bare metal (exponential filter, $\alpha \approx 0.0003$ at 192\,kHz).
  \item \textbf{Octave control}: 9 octaves (1--9), navigable via arrow keys.
  \item \textbf{Quick mode}: Fast attack/decay toggle for staccato playing.
  \item \textbf{Metronome}: Tempo-synced beat indicator (20--300 BPM), visual flash on downbeats.
\end{itemize}

\subsection{Input and hardware (2024--2026)}

\texttt{notepat} drove three hardware integration efforts:

\begin{enumerate}
  \item \textbf{NuPhy Air60 HE analog keyboard}: WebHID protocol reverse-engineered from USB captures. 64-byte activation commands, 16-bit pressure reports per key. Velocity-weighted synthesis. Required \texttt{CONFIG\_HIDRAW=y} in the bare-metal kernel.
  \item \textbf{Touchpad as expression controller}: X-axis controls reverb wet mix, Y-axis controls pitch shift. These mappings were designed specifically for \texttt{notepat}'s playing style.
  \item \textbf{HDMI secondary display}: Solid-color output driven by blended note colors. Added to \texttt{ac-native} in March 2026, motivated by \texttt{notepat}'s live performance use case.
\end{enumerate}

\subsection{Visual design}

The interface evolved from plain colored rectangles to a layered system:

\begin{itemize}
  \item 24-pad split grid (4$\times$3 left octave + 4$\times$3 right octave),
  \item Status bar: clock (LA time with DST), waveform, octave, sample rate, volume, battery, WiFi antenna icon,
  \item Mini piano keyboard visualization (24 keys with octave shading and black-key dots),
  \item HUD label in MatrixChunky8 font with \texttt{.com} superscript branding,
  \item Background tint reactive to time of day (dawn/midday/afternoon/sunset/night),
  \item Note color system: each semitone mapped to a hue (C=red through B=violet), active notes blend into background.
\end{itemize}

\subsection{Network and system integration}

\begin{itemize}
  \item WiFi UI: fullscreen network scanner with signal-strength bars, SSID picker, password entry mode---built into the piece itself for bare-metal use where no OS UI exists.
  \item Dark mode: automatic switch at 7\,pm/7\,am LA time, with matching DST calculation in both C and JavaScript.
  \item Projector and recital modes for performance contexts.
\end{itemize}

\section{The Bare-Metal Port}

In late 2024, \texttt{notepat} was ported to \texttt{ac-native}: a custom Linux runtime that boots from USB as PID~1 with no operating system. The bare-metal version (987 lines) runs on QuickJS, renders via DRM/KMS dumb buffers with 3$\times$ nearest-neighbor CPU upscale, and synthesizes audio through direct ALSA \texttt{hw:} access at 192\,kHz.

This port was not incidental. The entire \texttt{ac-native} project---custom kernel configuration, initramfs build pipeline, ALSA audio engine with 32-voice polyphony, evdev input handling, WiFi management via \texttt{wpa\_supplicant}---was built primarily to run \texttt{notepat} on physical hardware. The piece justified the platform.

The boot sequence itself reflects this: after kernel init, \texttt{ac-native} displays a rainbow-animated title, then loads \texttt{notepat.mjs} as the default piece. The instrument is playable within $\sim$2 seconds of power-on.

\section{Derivatives and Reuse}

Four artifacts derive directly from \texttt{notepat}:

\begin{enumerate}
  \item \textbf{\texttt{autopat.mjs}} (214 lines): Autoplay wrapper with a built-in jukebox (Twinkle, Bach excerpt, scales). Imports and delegates to \texttt{notepat}'s core lifecycle functions.
  \item \textbf{\texttt{notepat-tv.mjs}} (55 lines): UDP-driven remote visualizer for art installations. Receives note data and drives turtle graphics on a projected surface.
  \item \textbf{\texttt{notepat-convert.mjs}} (103 lines): Notation translation library. Converts single-letter keyboard shortcuts to extended melody syntax. Reused by \texttt{clock}, \texttt{stample}, and other music pieces.
  \item \textbf{Bare-metal \texttt{notepat.mjs}} (987 lines): Standalone reimplementation for the QuickJS runtime, sharing keyboard mappings and UI conventions but adapted for direct hardware access.
\end{enumerate}

\section{Infrastructure Pulled into Existence}

\subsection{Routing and branding}

The domain \texttt{notepat.com} is rewritten at the edge to the \texttt{notepat} slug in the primary index function. This required dedicated host-matching logic, custom Open Graph metadata, and a branded HUD element visible during play. The piece is the only one in the system with its own domain.

\subsection{Testing infrastructure}

\texttt{notepat} has disproportionate testing investment relative to other pieces:

\begin{itemize}
  \item \textbf{Latency harness} (\texttt{artery/test-notepat-latency.mjs}): Measures keyboard-to-sound latency via Chrome DevTools Protocol. Reports mean, median, p95, p99. Current measurement: 417\,ms (target: $<$30\,ms).
  \item \textbf{Stability harness} (\texttt{artery/test-notepat-stability.mjs}): 5-minute fuzz test at configurable intensity (2--20 notes/sec). Monitors heap growth, sound dictionary size, latency degradation. Latest result: 3{,}229 notes, $-$14\% heap, 0 warnings.
  \item 14 total test scripts across \texttt{artery/}, \texttt{tests/}, and \texttt{.vscode/tests/}.
\end{itemize}

No other piece in the system has dedicated performance testing.

\subsection{Piece-hit analytics}

The piece-hit tracking system (\texttt{/api/piece-hit}) was built in late December 2025. It records page loads server-side to MongoDB, distinguishing system and user pieces. In the initial telemetry window (2025-12-31 through 2026-01-02), \texttt{notepat} recorded 27 hits and ranked 4th among 44 tracked built-ins. The system tracks hits only in production (disabled in dev mode) and fires anonymously without auth headers. Current instrumentation limitations---no typed namespaces, no authenticated attribution, no long-horizon retention---mean that the full picture of \texttt{notepat} usage since 2024 is not captured.

\section{Political Economy of a Piece}

The trajectory of \texttt{notepat} illustrates feedback loops between a piece and its host platform:

\begin{enumerate}
  \item \textbf{Feature pressure $\rightarrow$ platform capability}: \texttt{notepat} needed room reverb, so the audio engine gained delay lines. It needed analog pressure, so \texttt{ac-native} gained HID raw support. It needed a secondary display, so HDMI output was added to the DRM backend.
  \item \textbf{Branding pressure $\rightarrow$ routing decisions}: The desire for a memorable URL led to \texttt{notepat.com}, which required edge-function host rewriting and custom social previews.
  \item \textbf{Quality pressure $\rightarrow$ testing investment}: A 417\,ms latency regression in \texttt{notepat} motivated the construction of a general-purpose piece-testing harness (artery), which now benefits all pieces.
  \item \textbf{Hardware pressure $\rightarrow$ entire runtime}: The ambition to run \texttt{notepat} on dedicated hardware justified building \texttt{ac-native}: custom kernel, initramfs, ALSA engine, DRM display, WiFi stack---thousands of lines of C.
\end{enumerate}

This is not unique to \texttt{notepat}. Any sufficiently invested-in piece will exert similar pressures. But \texttt{notepat}'s trajectory is unusually legible because it is well-documented and its 401-commit history is public.

\section{Conclusion}

\texttt{notepat} demonstrates that a single piece can function as the strategic center of gravity for a creative computing platform. Its growth from a 200-line keyboard sketch to a 9{,}000-line ecosystem with its own domain, hardware runtime, and test infrastructure was not planned---it emerged from sustained use and the feedback loops described above.

The methodological lesson is that studying platforms at the piece level reveals dynamics invisible at the system level: which features get built and why, which pieces attract disproportionate investment, and how branding and hardware decisions follow from interaction design rather than preceding it.

\begin{quote}
\small\noindent\textbf{摘要。}
关于软件系统的学术论文几乎只描述成功之处。本文描述的是休眠之处。在五年的开发过程中，\ac{}积累了16个残留功能——曾被积极开发、在一段时间内发挥了作用、然后归于沉寂的代码路径。如同生物学中的退化器官，这些功能并非死去的组织：有些可能在条件改变时重新激活，有些为存活下来的功能贡献了遗传物质，有些则因为有机体进化超越了它们而萎缩。我们将这些功能分为五类：\emph{平台分支}（在七个平台上同时开发，其中五个进入休眠）、\emph{基础设施实验}（分布式架构合并为更简单的形式）、\emph{商业化尝试}（技术上完整但等待市场条件的系统）、\emph{单用户工具}（服务于开发目的后被抛弃的功能）和\emph{过早抽象}（被特定解决方案取代的通用方案）。对于每一个，我们记录了休眠的原因、开发分歧的日期、分歧时的用户活动以及重新激活的可能性。我们认为，一个项目的残留功能——而非其失败——揭示了其进化轨迹，休眠并非死亡。
\end{quote}

\section{引言}

\ac{}的git历史包含跨越四个代码仓库的11,000多次提交~\citep{scudder2026archaeology}。已发表的论文描述了那些通往某处的提交：运行时、KidLisp解释器、piece系统、裸机操作系统~\citep{scudder2026ac, scudder2026os}。本文描述的是那些归于沉寂的提交。

软件进化，如同生物进化，通过分支进行。大多数分支不会成为主干。但与树上的枯枝不同，软件项目中休眠的代码路径可以重新激活：Android应用可能明天就上架；Electron应用可能找到使用场景；NFT基础设施已经超越了其最初的商业化目的，成为了piece-code寻址系统。\ac{}的不寻常之处在于维护着一个\texttt{GRAVEYARD.txt}文件，明确编目已弃用的功能——但"墓地"是个错误的比喻。更好的比喻是阑尾：一种原始功能已改变、以缩减形式存在、在不同条件下可能还有用的结构。

这份清单分为五类，每类代表一种不同\emph{类型}的休眠。这些类别并非互斥；有些功能属于多个类别。

\section{平台碎片化}
\label{sec:fragmentation}

\ac{}历史上最昂贵的死胡同类别是试图同时为每个平台构建原生应用。在2023年至2025年间，以下平台建立了可工作的原型：

\begin{itemize}
  \item \textbf{Web}（主要平台，始终可用）
  \item \textbf{Electron桌面应用}（macOS、Windows、Linux）
  \item \textbf{Android应用}（Kotlin + WebView，两个构建变体）
  \item \textbf{Apple TV应用}（计划中，从未开始）
  \item \textbf{Feral File FF1硬件}（FFOS集成）
  \item \textbf{FedAC信息亭}（Fedora + Electron信息亭模式）
  \item \textbf{AC Native OS}（裸机Linux）
\end{itemize}

在这七个目标中，目前只有两个处于活跃状态：Web平台和裸机操作系统。其他五个代表了大量工程投入，但未产生任何交付的产品。

\subsection{Electron应用}

一个完整的Electron应用已构建完成，包含macOS和Windows的代码签名证书、自定义终端集成、用于在piece和调试控制台之间切换的"翻转视图"，以及三个操作系统的构建基础设施。它从未分发给开发团队以外的任何用户。总投入——签名证书、构建脚本、CI管道、自定义JS桥接——产出了一个只在开发者机器上本地运行的应用。

\textbf{被保留的价值：}Electron的信息亭模式经验启发了FedAC信息亭设计，后者又启发了裸机操作系统方法。Electron应用证明了"窗口中的浏览器"是不够的——用户体验与打开浏览器标签页没有足够的差异来证明一个独立应用的合理性。

\subsection{Android应用}

一个完整的Android应用使用Kotlin和Jetpack Compose构建，包括两个构建变体（消费者版和信息亭版）、自动化GitHub Actions构建和发布APK。Google Play Console账户从未创建。应用从未提交到Play Store。

\textbf{被保留的价值：}信息亭变体的设备管理和锁定任务功能验证了单一用途AC设备的概念，这成为了\acos{}的设计目标。

\subsection{Feral File集成}

一个DP-1 feed协议客户端、播放列表中继代理和KidLisp-to-FF1投射机制被构建用于在Feral File的FF1硬件上运行\ac{} piece（约25台设备出货）。FFOS内核补丁已准备但从未合并。"Aesthetic Computer频道"启动器从未安装在任何FF1设备上。

\textbf{被保留的价值：}认识到修补他人的操作系统是一项结果不确定的"重型工程"。这直接推动了从头构建自定义操作系统，而非fork现有操作系统。

\subsection{模式}

每次平台尝试都遵循相同的弧线：构建一个可工作的原型，准备分发基础设施，然后在实际交付给用户的"最后一公里"停滞。失败不是技术性的——每个原型都能工作。失败是\emph{注意力}上的：一个开发者同时维护七个平台目标意味着没有一个能获得达到生产质量所需的持续关注。

Lehman的持续变化定律~\citep{lehman1980programs}预测了这一结果：一个系统必须不断适应才能保持令人满意。七个平台需要七条持续适应的流，这是一个独立开发者无法维持的。

\section{基础设施过度工程}

\subsection{Jamsocket会话服务器}

\ac{}的会话服务器最初部署在Jamsocket上，这是一个用于临时后端容器的平台。每个用户会话都会生成一个专用容器，拥有自己的WebSocket端点、Redis连接和生命周期管理。架构是优雅的：会话独立扩展，崩溃不影响其他用户，并且可以部署到不同区域。

它被一个单体服务器替代了。Jamsocket集成代码在一次提交中被移除。旧的EventSource轮询、API调用和扩展逻辑成为了死代码。

\textbf{失败原因：}管理临时容器的开销——冷启动延迟、连接切换复杂性、每会话成本——对于一个拥有数十个并发用户（而非数千个）的平台来说超过了收益。单体服务器以更低的复杂性处理了实际负载。

\textbf{被保留的价值：}会话服务器的内部架构（聊天管理、多人状态同步）在迁移中幸存下来。只有部署包装器被丢弃了。

\subsection{Bumper系统}

一个用于复杂会话/用户状态管理的"bumper系统"被构建，包括HUD元素和notepat集成。其确切用途现在即使对开发者来说也不清楚了。它在一个标题为"fix: completely remove all bumper system code"的提交中被移除。

\textbf{被保留的价值：}无。Bumper系统是项目中最纯粹的死胡同——构建、部署、移除，在当前代码库中没有留下任何痕迹。

\subsection{Silo迁移}

一个自托管的MongoDB实例（"Silo"）在DigitalOcean上配置，以降低相比MongoDB Atlas的成本。编写了安装文档，构建了仪表板，创建了基准测试。迁移从未执行。Silo droplet以每月12美元的价格运行着，空空如也，等待一次可能永远不会到来的迁移。

\textbf{成本：}每月12美元 $\times$ 6个月 = 72美元（且仍在增加），用于一个未使用的服务器。

\section{商业化实验}

\subsection{Tezos NFT（Keeps）}

\ac{}的KidLisp piece可以作为FA2 NFT在Tezos区块链上铸造。技术基础设施可以工作：SmartPy合约部署到主网和测试网、IPFS媒体存储、Objkt市场集成、钱包连接流程，9位所有者铸造了34个项目。

市场没有到来。KidLisp从未进入Tezos交易量前25的收藏（24小时内约72 XTZ，而门槛为100+ XTZ）。地板价稳定在每件4.42美元。更广泛的Tezos NFT市场在2025年持续萎缩，所有收藏的"大量供应过剩"抑制了买家注意力。

\textbf{被保留的价值：}KidLisp piece代码系统（如\texttt{\$cow}这样的短字母数字标识符）是为NFT铸造而设计的，但成为了通用的piece寻址方案。生成、存储和共享代码标识程序的基础设施超越了其最初的商业化目的。

\subsection{Sotce.net订阅}

一个订阅平台（sotce.net）使用Stripe集成、Auth0认证、页面编辑器和与\ac{}的跨租户账户同步构建。月订阅5美元。技术实现是完整的且可运行的。

订阅增长保持低迷。该平台产生的收入不足以证明其基础设施复杂性的合理性。它继续运行着。

\subsection{Printful商品}

一个Printful商家账户用于贴纸和马克杯。构建了一个"shop" piece。产品目录保持有限。收入可以忽略不计。

\textbf{模式：}所有三个商业化实验共享相同的结构：技术上完整，商业上惰性。工程不是瓶颈；分发、营销和市场契合度才是。

\section{为一个人构建的工具}

\subsection{录像机}

一个带有Instagram和Twitter质量预设的屏幕录制和导出工具被构建，开发者使用了一段时间，然后在"oven"烘焙系统取代它后被弃用。它现在存放在一个字面上名为\texttt{deprecated/}的目录中。

\subsection{Windows容器开发}

尝试通过Windows容器进行本地UDP开发以实现notepat同步。它"有时就是不工作。"整个Windows开发路径被放弃，转而只在Fedora上开发。

\subsection{\texttt{.pjs}编译器}

构想了一种"提示式JavaScript"文件格式：\texttt{.pjs}文件将在运行时发送给LLM，由LLM生成可执行的JavaScript。写了一条TODO。没有代码随后产生。该格式仍然是\texttt{GRAVEYARD.txt}中的一条注释。

\textbf{模式：}这些功能中的每一个都是为了解决只有开发者自己遇到的问题而构建的。当开发者的工作流程改变时，功能就过时了。这是独立开发的职业风险：用户群是$n=1$，当$n=1$转移了，功能就有零个用户。

\section{过早抽象}

\subsection{通用信息亭模式}

尝试构建一个统一的"信息亭模式"抽象，可在Android（设备管理+锁定任务）、Electron（全屏+禁用菜单）和Fedora（Wayland合成器锁定）上工作。每个平台的信息亭需求差异足够大，以至于抽象没有提供任何好处。统一方法被放弃，转而采用平台特定的实现。

\textbf{被保留的价值：}信息亭计算需要硬件特定的解决方案，而非通用抽象的洞察。这一洞察直接导致了裸机操作系统方法，该方法不尝试在硬件之间进行抽象。

\subsection{FFOS Fork}

评估了fork Feral File的操作系统以创建通用x86\_64信息亭的方案，并因其为"重型工程"而被拒绝。FFOS架构对FF1硬件过于专业化（ARM、特定显示管线、自定义引导程序），无法推广到商品化笔记本电脑。

\textbf{被保留的价值：}\emph{不}fork FFOS的决定迫使从头创建\acos{}，这产生了一个比fork更简单、更易维护的系统。

贡献率因类别而差异巨大。过早抽象的贡献率为100\%，因为它们的休眠产生了\emph{信息}——具体来说，即通用解决方案不合适的知识。平台分支的贡献率为80\%，因为每个休眠的平台都教会了存活平台需要什么。

十六个功能中有七个可以复活：Android应用（Play Store提交仍然可能）、Electron应用（桌面发布可以服务特定用例）、Feral File集成（如果FF1硬件规模扩大）以及所有三个商业化实验（Tezos Keeps、Sotce.net、商品——每个都技术完整，等待市场条件而非工程工作）。Silo迁移如果MongoDB Atlas成本变得不可持续，可以在一天内执行。

单用户工具的贡献率和复活率均为0\%。它们是开发脚手架——在建设过程中有用，之后被移除，没有留下痕迹。

\subsection{休眠揭示了什么}

一个项目的残留功能揭示了它在成为现在之前\emph{试图成为什么}——以及如果条件改变它可能还会成为什么。\ac{}的休眠路径揭示了：

\begin{itemize}
  \item 渴望\emph{无处不在}（七个平台），被规训为\emph{在特定地方}（Web + 裸机）——但Android和Electron代码仍然可以编译
  \item 相信\emph{技术完整性}（可工作的原型）等同于\emph{产品就绪}（交付给用户）——被证伪但未被否定，因为原型仍然可以运行
  \item 假设\emph{构建商业化基础设施}就会吸引市场——市场还没有到来，但如果它到来，基础设施是准备好的
  \item 偏好\emph{从头构建}而非适应现有系统，在FFOS fork被拒绝、转而构建自定义操作系统时得到了验证
\end{itemize}

\section{结论}

五年内十六个残留功能。一半为存活功能贡献了代码或洞察；七个仍可复活。生物学比喻比"失败"更诚实：这些不是死去的组织，而是活体系统中的休眠结构。Android应用没有失败——它睡着了。NFT基础设施没有失败——它脱落了商业化功能，保留了寻址方案。Jamsocket架构没有失败——它把内部结构贡献给了取代它的单体服务器。

\ac{}历史上最重要的休眠路径是产生了最有价值结果的那条：构建通用信息亭抽象的尝试进入休眠，迫使从头创建裸机操作系统——现在是项目技术上最具特色的贡献。最好的残留功能是其休眠使有机体构建出正确肢体的那个。

Brooks观察到"计划扔掉一个；无论如何你都会"~\citep{brooks1975mythical}。\ac{}搁置了十六个。七个可能明天就醒来。第十七次尝试——一个在7.3秒内启动的单文件操作系统——仍在运行。
